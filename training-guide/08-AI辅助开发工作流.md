# AI 辅助开发工作流

## AI 最适合做什么

让 AI 帮你读代码、定位调用链、解释错误、拟定小改动、生成测试、审查差异和补充文档。把 AI 当结对开发者，不当最终审批者；运行结果、权限和数据安全由人负责。

## 推荐六步法

1. **给目标**：用户场景、期望结果、不做什么。
2. **给证据**：相关文件、完整错误、复现步骤；用脱敏内容。
3. **先调查**：要求 AI 先读 README、项目约束、相关调用链和 `git status`。
4. **小步修改**：限制目录和范围，不顺手重构无关代码。
5. **自动验证**：让 AI 运行 `pnpm build`、`mvn test` 等，并报告原始失败。
6. **人工验收**：自己检查 diff，在浏览器走关键流程，再提交 PR。

## 可直接复用的提示词

### 定位错误

```text
请先不要改代码。阅读与【功能】有关的前端 API、页面、后端 Controller/Service/Repository，
根据下面的复现步骤和完整报错定位根因。请区分事实与猜测，给出最小修复方案和验证方法。
复现：...
期望：...
实际：...
报错（已脱敏）：...
```

### 实现小功能

```text
在当前仓库实现【功能】。范围只限【目录/模块】，保持现有 Vue 3 + TypeScript 和
Spring Boot 分层风格。先检查 git status，保留已有改动；不要修改生产配置或真实数据。
完成后运行相关构建/测试，并总结改动、风险、手工验证步骤。
验收条件：1... 2... 3...
```

### 审查代码

```text
请审查当前分支相对 main 的差异，不要修改。重点检查权限绕过、数据破坏、上传安全、
空值/边界、前后端契约、数据库兼容和缺失测试。按严重程度列出具体文件与原因。
```

### 解释命令

```text
请逐段解释这条命令会读写哪些文件、是否访问网络/生产环境、能否回滚。
如果包含删除、覆盖、数据库写入、sudo 或强制 Git 操作，先停止并给安全替代方案。
命令：...
```

## 给 AI 的上下文清单

- 当前分支与 `git status --short`。
- 任务涉及页面/API/表名和准确验收条件。
- 软件版本：Node 20、pnpm 10.30.0、Java 17、Spring Boot 3.5.3、MySQL 8。
- 环境是本地、测试还是生产（生产操作必须明确授权）。
- 已尝试什么及完整输出，不只截最后一行。

## 绝不能发给 AI 的内容

数据库/邮箱密码、SSH 私钥、GitHub Token、Cookie/JWT、包含个人资料的数据库导出、未公开域名/IP 策略。用 `<DB_PASSWORD>`、`example.com` 和脱敏样例替代；一旦误发，应立即轮换秘密，而不是只删除聊天。

## 识别 AI 的危险答案

- 没读代码就假定框架/路径。
- 建议关闭安全检查、放行全部权限或把密码写死。
- 使用 `rm -rf`、`git reset --hard`、`push --force`、无 WHERE 的 UPDATE/DELETE。
- 只说“应该可用”却没运行构建测试。
- 为消除类型错误大量使用 `any`，或为过测试删除测试。
- 同时改很多无关文件，让结果无法审查。

遇到这些情况应停止，让 AI 解释影响并给最小、可回滚方案，必要时请有经验的维护者复核。
