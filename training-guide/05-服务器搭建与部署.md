# 服务器搭建与部署

> 本章涉及生产状态变化，仅限获得授权的维护者操作。新人先在测试服务器演练。仓库自动化目标是 Ubuntu 22.04。

## 生产结构

| 内容 | 当前路径/服务 |
|---|---|
| 后端 JAR | `/opt/webserver/app/webpage-0.0.1-SNAPSHOT.jar` |
| 后端备份 | `/opt/webserver/backup`，工作流保留最新 6 个 |
| 前端发布 | `/var/www/CDMGA-vue/release_main`、`release_dev` |
| 前端当前版本 | `/var/www/CDMGA-vue/current_main`、`current_dev` 软链接 |
| 上传文件 | `/var/webserver/{images,scores,avatars,banners,files}` |
| 后端守护 | `webserver.service`，用户 `webserver` |
| Nginx 配置 | `/etc/nginx/sites-available/cdmga_nginx.conf` |
| 后端日志 | `journalctl -u webserver.service` |

## 上线前的基础设施

- 使用普通 sudo 运维账户和 SSH 密钥，禁止密码弱口令；云安全组只开放需要的 22（或自定义 SSH）、80、443。
- MySQL 3306 不对公网开放；应用与数据库同机时绑定本机，异机时仅允许应用服务器私网 IP。
- 配置域名和 HTTPS（仓库当前 Nginx 配置只监听 80，生产应增加证书与 HTTP→HTTPS 跳转）。
- 建立数据库和上传文件的异机备份，并定期做恢复演练。
- 服务器时间区按业务统一，开启安全更新、磁盘与服务监控。

## 安装与目录

仓库 `.github/workflows/init.yml` 会通过 SSH 安装 Java 17、Nginx，创建 `webserver` 用户和目录，并继续部署环境、后端和前端。首次执行前应人工审查脚本和目标服务器；当前 init 最后仍调用已废弃的 `npm_update.yml`，而正常发布使用 `pnpm_update.yml`，首次初始化前应修正这一不一致。

## 生产配置与密钥

GitHub 仓库 Actions Secrets 需要：`SSH_HOST`、`SSH_SUDO`、`SSH_PRIVATE_KEY`、`SSH_PORT`、`SSH_DOMIAN`。注意仓库沿用拼写 `DOMIAN`，不要擅自只改一处。

systemd 会读取 `/etc/webserver.env`。推荐让 `application.properties` 引用环境变量，然后在服务器保存：

```bash
DB_URL=jdbc:mysql://127.0.0.1:3306/CDMGA?useSSL=false&serverTimezone=Asia/Shanghai
DB_USER=<应用专用低权限用户>
DB_PASSWORD=<强密码>
```

文件权限建议 `root:webserver` 且 `640`。不要把这个文件上传 Git 或发给 AI。修改后 `sudo systemctl restart webserver.service`。

## 自动部署行为

- 推送 `main`：`CD.yml` 依次更新环境，再并行/后续部署后端和 main 前端。
- 推送 `dev`：`CD-dev.yml` 更新环境并只部署 dev 前端；当前不会部署 dev 后端，两站仍共用 8080 后端。
- 后端流程构建 `-P prod`，备份旧 JAR，启动 systemd，最多等待约 3 分钟检测 8080。
- 前端流程用 pnpm 10.30.0 + Node 20 构建，将时间戳目录发布并原子切换软链接。

因此合并到 `main` 就是生产发布动作。必须启用分支保护、PR 审批和 Actions 环境审批，不能让新人直接 push。

## 手工检查命令

```bash
sudo systemctl status webserver.service --no-pager
sudo journalctl -u webserver.service -n 100 --no-pager
sudo nginx -t
sudo systemctl status nginx --no-pager
curl -i http://127.0.0.1:8080/hello
curl -I http://127.0.0.1/
df -h
```

修改 Nginx 必须先 `nginx -t` 成功再 reload：`sudo systemctl reload nginx`。

## 回滚

前端：找到已验证的 `/var/www/CDMGA-vue/release_main/<时间>`，把 `current_main` 软链接原子指回该目录，再访问关键页面。后端：停止服务，将 `/opt/webserver/backup` 中确认的旧 JAR 复制回固定 app 路径，校正所有者/权限，再启动并检查日志。数据库不一定与旧代码兼容，所以任何数据库变更都必须另有回滚方案；不要假设换回 JAR 就完成回滚。

## 还需补齐的生产能力

HTTPS、安全认证、数据库迁移、健康检查（Spring Actuator）、集中日志/告警、备份恢复演练、依赖漏洞扫描、上传文件安全扫描。这些是上线质量工作，不建议让新人凭 AI 一次性改完。
