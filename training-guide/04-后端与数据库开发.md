# 后端与数据库开发：Spring Boot + MySQL

## 分层职责

- Controller：解析 HTTP 参数并返回结果，不堆积复杂业务。
- Service：业务校验、流程和事务。
- Repository：JPA 查询与数据库访问。
- Entity：数据库表映射；改变字段可能需要数据库迁移。
- DTO/common：对外输入输出，避免直接暴露 Entity 的所有字段。

新增功能通常按“表/迁移 → Entity/DTO → Repository → Service → Controller → 前端 API”完成。不要让 Controller 直接拼 SQL。

## 当前配置和运行

后端端口是 8080，Java 17，默认 Maven Profile 为 `dev`。生产包：

```powershell
cd webpage
.\mvnw.cmd clean package -P prod
```

产物为 `target/webpage-0.0.1-SNAPSHOT.jar`。Linux 对应使用 `./mvnw`。

## API 约定

多数接口位于 `/api/*`，通用响应类是 `common/Result.java`。开发新接口时：

- 使用符合含义的 HTTP 方法：查询 GET，创建 POST，完整/主要更新 PUT，删除 DELETE。
- 对请求 DTO 添加 Jakarta Validation，并在 Controller 用 `@Valid`。
- 给 400、401、403、404、409、500 清晰且不泄露内部信息的响应。
- 分页统一 `page`、`pageSize/size` 约定；当前项目两种名字并存，新增代码应选定一种并逐步统一。
- 不把密码、数据库异常、服务器路径返回前端。

## 数据库变更

生产安全流程：

1. 先写可审查、可重复执行策略清晰的迁移脚本。
2. 在本地脱敏副本执行并验证。
3. 备份生产库，并实际验证备份可恢复。
4. 评估锁表、字段默认值、旧代码兼容性。
5. 先发布兼容数据库变更，再发布代码；删除字段放在后续版本。
6. 记录执行时间、操作者、脚本和回滚办法。

不要直接把 `CDMGA.sql` 导入日常开发库：它包含 `DROP TABLE` 和实际记录。`ddl-auto=update` 适合早期本地开发但不适合可控生产迁移，建议引入 Flyway 后将生产改为 `validate`。

## 文件上传

生产文件由后端写入 `/var/webserver/images`、`scores`、`avatars`、`banners`、`files`，Nginx 通过同名 URL 前缀读取。新增上传类型时要同步修改：属性、目录初始化、systemd `ReadWritePaths`、Nginx alias、备份策略。

必须校验扩展名与真实 MIME、随机化服务端文件名、防止 `../` 路径穿越、限制大小、禁止可执行文件，并设置下载响应头。目前 Nginx 5 MB 与 Spring 10 MB 不一致，应统一。

## 身份与权限（当前最高风险）

代码虽然引入 Spring Security，但 `SecurityConfig` 放行所有路径；登录信息主要是前端 session 状态。这意味着知道接口的人可能绕过管理页面直接请求删除、审核等接口。

整改建议按优先级：

1. 后端建立服务端 Session 或现代 JWT 认证过滤器，密码仅用 BCrypt/Argon2 哈希。
2. `/api/identity/login`、注册和确需公开的 GET 明确放行，其余默认认证。
3. 管理接口用 `hasRole("ADMIN")` 或方法级 `@PreAuthorize`。
4. 所有“操作自己数据”的接口从认证主体取用户 ID，不信任前端传来的 `identityId`。
5. 添加未登录 401、普通用户 403、越权访问和文件上传的自动测试。

## 最低测试

```powershell
cd webpage
.\mvnw.cmd test
```

当前测试很少。每个修复至少补 Service 单元测试或 Controller 集成测试；涉及数据库可使用专用测试库/Testcontainers，绝不能对生产库跑测试。重点覆盖成功、空值、边界、重复提交、不存在记录、普通用户越权和数据库失败。
