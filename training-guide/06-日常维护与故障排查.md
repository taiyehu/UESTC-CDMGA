# 日常维护与故障排查

## 每日/每次发布后

- 打开首页、登录、目标业务页和管理页。
- 查看 GitHub Actions 是否全部绿色；失败时先读第一个失败步骤。
- 检查 `systemctl status`、最近后端/Nginx 错误日志、磁盘空间。
- 对新增/修改功能做一条成功路径和一条失败路径。
- 确认上传图片可访问、深层链接刷新不 404。

## 标准排错法

1. 停止重复点击和随机改配置。
2. 写清“期望什么、实际什么、何时开始、影响谁、刚改过什么”。
3. 保留完整错误，从最早出现的异常和最后一个 `Caused by` 看起。
4. 定位层次：浏览器 → Nginx → 后端 → MySQL/磁盘。
5. 一次只验证一个假设；修复后复测原问题和相邻功能。
6. 记录根因和解决方案，必要时补测试。

## 状态码速查

| 状态 | 常见含义 | 先检查 |
|---|---|---|
| 400 | 参数格式/校验失败 | Network 请求体、后端 DTO |
| 401 | 未登录 | 会话/Token（完成后端认证后） |
| 403 | 已登录但无权限 | 用户角色与后端授权规则 |
| 404 | 地址或数据不存在 | 路由、Controller、Nginx location |
| 413 | 上传过大 | Nginx `client_max_body_size` |
| 500 | 后端异常 | `journalctl` 最早异常 |
| 502 | Nginx 连不上后端 | 后端服务、8080、proxy_pass |
| 504 | 后端处理超时 | 数据库慢查询、死锁、服务负载 |

## 典型故障

### 网站完全打不开

查 DNS/证书/安全组，再查 Nginx 状态和错误日志。只有域名失败而 IP 成功，多为 DNS/证书；80/443 都拒绝，多为 Nginx 或防火墙。

### 页面能开但所有数据失败

在 Network 看 `/api` 状态。502 查 Spring 服务；500 查后端/数据库；404 查 Nginx 是否保留 `/api/` 前缀。本项目 `proxy_pass http://localhost:8080` 会保留原 URI。

### 后端启动失败

```bash
sudo journalctl -u webserver.service -n 200 --no-pager
sudo ss -lntp | grep :8080
```

常见原因：数据库不可达/密码错、JAR 不存在或权限错、上传目录不可写、8080 被占、配置变量缺失。

### 上传失败或图片裂开

依次检查浏览器请求、413/500、磁盘是否满、`/var/webserver` 权限、systemd `ReadWritePaths`、Nginx alias 和文件是否真实存在。数据库只保存 URL，上传文件必须单独备份。

### 发布后白屏

查浏览器 Console、前端构建是否成功、`current_main` 软链接、静态文件权限。可先回滚前端软链接，再分析失败版本。

### 数据异常

先停止会扩大损害的写操作，备份现状并保留日志。不要直接执行 AI 生成的 UPDATE/DELETE。用 SELECT 确定影响行数，让第二人复核事务脚本，再决定修复。

## 备份对象

- MySQL 全库（加密、异机保存、验证恢复）。
- `/var/webserver` 的全部上传文件。
- `/etc/webserver.env` 与 Nginx/systemd 配置（密钥单独安全保存）。
- 应用版本由 Git tag/commit 和发布记录追踪，不用长期堆积无限 JAR/dist。

任何备份只有成功恢复过才算可靠。
